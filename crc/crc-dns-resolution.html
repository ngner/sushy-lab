Resolution Steps — google.com

1App calls getaddrinfo("google.com") → glibc NSS
2glibc reads /etc/resolv.conf → UDP to systemd-resolved stub at 127.0.0.53
3resolved checks per-link routing domains: "google.com" does NOT match ~testing on crc
4Selects enp198s0f3u1 — the only link with DefaultRoute: yes
5Forwards UDP:53 query to 192.168.168.1 (LAN router) via enp198s0f3u1
6Response returns same path → app gets IP address. CRC never involved.

Resolution Steps — console-openshift-console.apps-crc.testing

1App calls getaddrinfo("...apps-crc.testing") → glibc NSS
2glibc → UDP to systemd-resolved stub at 127.0.0.53
3resolved checks per-link routing: "*.testing" MATCHES ~testing on crc link
4Selects crc bridge — configured by 99-crc.sh NM dispatcher (resolvectl dns crc 192.168.130.11)
5Forwards UDP:53 to CRC VM at 192.168.130.11 via crc bridge (192.168.130.1/24)
6CRC VM's CoreDNS resolves the OpenShift route → returns 192.168.130.11

CRC Stopped — Impact Summary

✓Normal queries (google.com, etc.) → enp198s0f3u1 → 192.168.168.1 — completely unaffected
✗.testing queries → crc link → 192.168.130.11 — timeout → SERVFAIL (VM not running)
•The crc bridge persists because the libvirt crc network stays active after crc stop
•Per-link DefaultRoute: no means resolved never sends non-.testing queries to this link
•No global drop-in files — nothing to leak. Compare with the old approach where DNS=192.168.130.11 in a global drop-in caused ALL queries to reach the CRC IP
